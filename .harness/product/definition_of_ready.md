# ✅ CONTRACT — Definition of Ready

A feature is "ready for Engineering" only when it meets ALL of these criteria.

---

## What is the Definition of Ready?

It is the minimum standard that a product brief must meet before Engineering can begin working on it. It is not optional. There are no exceptions.

If a backlog item does not meet the Definition of Ready, it returns to the PM — it does not enter the sprint.

---

## Checklist — Definition of Ready

### Problem Clarity
- [ ] The problem it solves is described in user terms, not system terms
- [ ] The target user is specific (a profile, not "everyone")
- [ ] There is evidence of pain: real feedback, abandonment metric, or explicit request

### Scope Definition
- [ ] The user story is written in standard format (As a / I want / so that)
- [ ] Functional acceptance criteria are complete and verifiable
- [ ] What is NOT included in this feature (scope boundary) is explicit
- [ ] Dependencies are identified (even if Engineering resolves them)

### Design and Flow
- [ ] The user flow is designed and approved by the PM
- [ ] UX Writing (button labels, messages, errors) is specified
- [ ] Conceptual validation by Product QA is completed and approved

### Metrics
- [ ] The post-launch success metric is defined and measurable
- [ ] It is clear how it will be measured (which event, which tool)

### Approvals
- [ ] CPO approved the PRD
- [ ] PM signed the handoff Product Brief

---

## What the Definition of Ready is NOT

- It is not a guarantee that the feature is technically trivial
- It is not a complete visual design (detailed wireframes are optional, not required)
- It does not replace the technical questions that Engineering will resolve during implementation

---

## If the feature does not meet the Definition of Ready

| Missing Criterion | Action |
|---|---|
| Problem not clear | PM returns to the CPO to redefine |
| Flow not designed | UI/UX completes the flow before moving forward |
| No acceptance criteria | PM completes them — Engineering cannot make assumptions |
| No CPO approval | The brief does not leave the Product Harness |
